# 全市能源审计平台重构设计文档

## 1. 文档目的

本文档用于固化“全市能源审计平台重构”项目的业务设计结论，作为后续完整开发、接口预留、实施排期和验收定义的统一依据。

本文档基于已确认的约束整理：

- 项目按“完整版”目标设计，不只覆盖一期最小闭环
- 总体设计采用“平台核心 + 业务模块化”方式
- 角色体系固定为三端：企业端、管理端、审核端
- 技术路线暂不锁定，先完成业务架构、数据模型、流程状态和实施路线设计
- 权限与企业主信息需预留对接“企业基本信息接口”，支持反向维护

## 2. 产品定位与系统边界

本系统应定义为“全市重点用能企业能源审计管理平台”，不是单纯的在线填报系统。

平台面向全市工业通信业重点用能单位，服务于十五五期间能源审计工作的组织、填报、审核、整改和管理决策。平台通过采集企业能耗、组织碳、产品单耗、设备能效、碳足迹、节能改进措施等数据，支撑自动生成审计报告、组织审核评分、形成整改闭环，并为主管部门提供精准推动节能改造与能效提升的数据基础。

平台完整闭环如下：

`企业注册/纳管 -> 企业基础配置 -> 审计数据填报 -> 系统计算与图表输出 -> 审计报告生成/上传 -> 审核评分与问题认定 -> 整改任务跟踪 -> 管理统计与决策支撑`

系统边界内包含三类核心能力：

- 数据采集能力
- 业务闭环能力
- 分析与产出能力

系统边界外内容先按“可对接对象”处理，不纳入当前设计内建范围：

- 政府外部主数据接口
- 第三方在线文档或 Office 协同
- 外部标准库或行业对标库
- 消息、短信或统一认证平台

## 3. 总体业务架构

### 3.1 分层结构

系统建议拆分为三层：

- 平台核心层
- 业务能力层
- 输出与监管层

### 3.2 平台核心层

平台核心层承载全平台通用能力：

- 组织与身份
- 权限与角色
- 字典与主数据
- 模板与版本
- 流程与状态机
- 审计留痕
- 附件与文件
- 消息与待办

### 3.3 业务能力层

业务能力层拆为 8 个核心业务域：

- A. 企业准入与企业管理
- B. 基础配置中心
- C. 审计数据采集中心
- D. 计算与校验中心
- E. 图表与分析中心
- F. 审计报告中心
- G. 审核评分中心
- H. 整改跟踪中心

### 3.4 输出与监管层

管理端内部建议拆分两类视图：

- 业务作业视图
- 统计决策视图

### 3.5 三端职责边界

企业端负责：

- 维护企业基础信息
- 配置能源、产品、单元
- 填写审计数据
- 查看校验结果
- 生成、下载、上传报告
- 接收审核意见
- 处理整改任务

管理端负责：

- 企业准入与纳管
- 账号与权限
- 基础模板和字典
- 填报任务管理
- 审核分派
- 报告归档
- 整改督办
- 统计分析

审核端负责：

- 查看被分派审核任务
- 查看企业填报数据与报告
- 按评分表评分
- 登记问题和建议
- 输出审核结论

## 4. 核心业务流程

### 4.1 企业准入流程

`企业申请 -> 管理端审核 -> 建立企业档案 -> 开通企业账号/关联身份 -> 生效`

说明：

- 企业基础身份信息优先来自外部企业基本信息接口
- 平台保存业务侧企业档案，用于审计资格、状态和联系人维护
- 外部接口未就绪时允许临时建档，后续再进行映射归并

### 4.2 审计任务启动流程

`管理端创建审计批次 -> 选择企业范围 -> 下发填报任务 -> 企业进入待填报`

顶层业务对象必须是“审计批次”和“企业审计项目”，后续报告、审核、整改均挂在企业审计项目下。

### 4.3 企业基础配置流程

`企业进入任务 -> 维护企业设置/能源/单元/产品 -> 完成主数据准备`

企业设置、能源设置、产品设置和单元设置属于前置主数据，不应被视为普通辅助页面。

### 4.4 审计数据填报流程

`企业录入/导入数据 -> 系统自动计算与校验 -> 企业修正 -> 提交`

系统需支持表单、表格、多维表、导入模板和关系型图式输入。提交动作必须经过完整性检查、逻辑校验和关联校验。

### 4.5 报告生成与归档流程

`填报完成 -> 系统生成报告初稿 -> 下载修订/在线编辑 -> 上传终稿 -> 归档`

报告至少区分系统初稿、企业终稿和归档终稿。

### 4.6 审核评分流程

`管理端分派审核任务 -> 审核端查看材料 -> 按评分表评分 -> 输出问题与结论 -> 管理端确认`

审核过程必须结构化留痕，不能只有通过或驳回结果。

### 4.7 整改跟踪流程

`从审核问题/节能潜力生成整改项 -> 企业填报整改计划与进度 -> 管理端跟踪 -> 完成闭环`

整改项应为独立业务对象，不只是报告里的文字描述。

### 4.8 管理统计与决策支撑流程

`平台汇总批次/企业/行业数据 -> 形成统计图表与台账 -> 支撑主管部门决策`

## 5. 核心业务对象与数据模型

### 5.1 平台核心对象

- `Enterprise`：企业主档案
- `EnterpriseExternalBinding`：企业与外部系统映射关系
- `UserAccount`：平台账号对象
- `Role` / `Permission`：角色与权限模型
- `DictionaryItem`：字典项对象
- `Template`：模板主对象
- `TemplateVersion`：模板版本对象
- `Attachment`：统一附件对象
- `AuditLog`：审计日志对象

### 5.2 业务运行对象

- `AuditBatch`：审计批次
- `AuditProject`：企业参与某批次的一次审计项目
- `ProjectMember`：项目参与关系
- `EnterpriseProfile`：审计期企业信息快照
- `EnergyDefinition`：企业级能源品种定义
- `ProductDefinition`：企业级产品定义
- `UnitDefinition`：企业级单元定义
- `DataModule`：填报模块定义
- `DataRecord`：填报记录主对象
- `DataItem`：字段或单元级数据对象
- `ImportJob`：导入任务对象
- `ValidationResult`：校验结果对象
- `CalculationSnapshot`：关键计算结果快照

### 5.3 结果输出对象

- `Report`：报告主对象
- `ReportVersion`：报告版本对象
- `ReviewTask`：审核任务对象
- `ReviewScore`：审核评分对象
- `ReviewIssue`：审核问题对象
- `RectificationTask`：整改任务对象
- `RectificationProgress`：整改进度对象
- `ChartOutput`：图表输出对象

### 5.4 关键关系

业务主链建议固定为：

`Enterprise -> AuditProject -> DataRecord -> Report / ReviewTask / RectificationTask`

配置对象支撑关系建议为：

`TemplateVersion -> DataModule -> ValidationRule / CalculationRule / ReportSection`

### 5.5 建模原则

- 配置数据与业务数据必须分层
- 历史项目必须绑定当时的模板版本
- 外部主数据变化不得静默覆盖历史项目快照
- 原始值、系统计算值、人工修正值、最终采用值应区分保存

## 6. 状态机与流程控制

### 6.1 企业准入状态

建议状态：

- `待审核`
- `已通过`
- `已驳回`
- `已停用`
- `已锁定`
- `已过期`

### 6.2 审计项目状态

建议状态：

- `待启动`
- `配置中`
- `填报中`
- `待提交`
- `待生成报告`
- `报告处理中`
- `待审核`
- `审核中`
- `待整改`
- `整改中`
- `已完成`
- `已关闭`

### 6.3 填报记录状态

建议状态：

- `草稿`
- `已保存`
- `校验失败`
- `待提交`
- `已提交`
- `已退回`
- `已归档`

### 6.4 报告状态

建议状态：

- `未生成`
- `已生成初稿`
- `企业修订中`
- `待提交终稿`
- `终稿已上传`
- `审核中`
- `已归档`
- `已作废`

### 6.5 审核任务状态

建议状态：

- `待分派`
- `已分派`
- `审核中`
- `待确认`
- `已退回`
- `已完成`
- `已关闭`

### 6.6 整改任务状态

建议状态：

- `待下发`
- `待认领`
- `整改中`
- `待验收`
- `已完成`
- `延期中`
- `已关闭`

### 6.7 全局推进规则

- 基础配置未完成，不允许进入完整填报
- 存在关键校验错误，不允许项目提交
- 未生成或未上传有效报告，不允许进入正式审核
- 审核结论存在问题项时，项目不能直接完成
- 整改任务全部完成后，项目才能闭环完成

## 8. 配置化设计原则

### 8.1 配置化总策略

建议采用：

- 主体业务流程代码化
- 字段结构和计算规则配置化
- 报告和图表模板化

### 8.2 配置化范围

建议配置化内容：

- 填报模块启停
- 字段定义
- 字段分组与布局
- 字段与字典绑定
- 必填与显示隐藏规则
- 简单联动规则
- 公式计算规则
- 单位换算规则
- 图表生成规则
- 报告章节与字段映射
- 导入模板格式
- 评分表结构

不建议早期高度配置化的内容：

- 登录认证流程
- 权限主逻辑
- 审计项目状态机
- 审核与整改主流程
- 文件归档主逻辑
- 外部接口同步框架

### 8.3 核心配置对象

- `ModuleConfig`
- `FieldConfig`
- `FieldOptionSource`
- `ValidationRule`
- `CalculationRule`
- `ChartConfig`
- `ReportTemplate`
- `ReviewTemplate`

### 8.4 配置优先级

建议支持以下覆盖层级：

`企业特例 > 企业类型 > 批次模板 > 平台默认`

### 8.5 在线表格定位

在线表格适合承载：

- 复杂二维或多维录入表
- 带公式的业务表
- 模板导入导出
- 类 Excel 展示和批量维护

在线表格不适合承载：

- 主流程控制
- 权限和状态控制
- 审核任务和整改闭环
- 报告版本管理

### 8.6 标签化和结构化抽取

建议建立稳定字段编码体系，所有模板单元格或区域均绑定业务编码，报告、图表和导入映射均基于业务编码，而不是按表格位置硬编码。

## 9. 数据校验、计算与规则体系

### 9.1 规则总目标

规则体系必须同时解决：

- 能不能填
- 填得对不对
- 算得出来吗
- 前后是否一致

### 9.2 校验规则分层

- 基础校验
- 字段逻辑校验
- 跨字段校验
- 跨模块校验
- 业务完整性校验

### 9.3 计算规则分层

- 字段级计算
- 模块级计算
- 项目级计算
- 输出级计算

### 9.4 一致性规则

- 主数据一致性
- 时间一致性
- 口径一致性
- 能源平衡一致性
- 报告一致性

### 9.5 执行时机

- 录入时
- 保存时
- 提交时
- 生成输出时

### 9.6 执行结果结构

校验结果建议统一返回：

- 规则编码
- 规则类型
- 所属模块
- 关联字段
- 严重程度
- 错误信息
- 修复建议
- 是否阻断提交

严重程度建议分为：

- `错误`
- `警告`
- `提示`

### 9.7 例外处理

建议支持“带说明提交”的受控例外机制：

- 管理端控制是否允许例外提交
- 企业端填写例外说明
- 审核端可查看全部例外记录
- 全流程保留审计痕迹

### 9.8 关键计算原则

涉及综合能耗、当量值、等价值、单位产值综合能耗、产品单耗、碳排放、节能量、节能潜力等指标时，应遵循：

- 基于稳定字段编码计算
- 参数与模板版本绑定
- 关键结果保留快照
- 自动计算值不得被静默覆盖

## 10. 报告生成、图表输出与文档产出

### 10.1 文档产出体系

平台结果输出建议拆为四类：

- 审计报告
- 图表输出
- 审核文档
- 整改台账

### 10.2 报告版本

报告至少分为：

- 系统初稿
- 企业修订稿
- 归档终稿

### 10.3 报告内容来源

报告内容来源分为：

- 直接字段映射
- 计算结果映射
- 图表嵌入
- 人工补充内容

原则：

凡可结构化的数据，不允许在报告中重复手填作为主来源。

### 10.4 报告模板能力

报告模板应支持：

- 章节结构
- 标题与顺序
- 字段映射
- 图表插入规则
- 表格映射规则
- 文本模板占位符
- 可编辑区域定义
- 版本管理

### 10.5 图表输出

图表应区分：

- 规定图表
- 辅助图表

图表来源统一来自标准汇总结果或计算快照。图表配置建议至少包含编码、名称、类型、维度字段、指标字段、适用模板版本、适用角色、导出能力和报告嵌入标记。

### 10.6 审核与整改文档

建议标准化输出：

- 审核评分表
- 审核意见汇总
- 问题清单
- 整改建议清单
- 整改台账

## 11. 外部系统对接与接口预留

### 11.1 总原则

- 外部系统负责身份与企业主数据来源
- 本平台负责审计业务状态
- 历史业务数据不随外部主数据静默漂移
- 接口失败不能直接阻断主业务链路

### 11.2 企业基本信息接口

接口预留目标：

- 获取企业主档基础信息
- 获取企业组织身份或账号映射信息
- 获取企业状态变化
- 支持本平台回写补充信息或标记信息
- 支持企业与外部主档绑定关系确认

### 11.3 反向维护边界

字段应拆分为三类：

- 外部主控字段
- 平台补充字段
- 协同字段

这样避免双向主数据混战。

### 11.4 绑定机制

企业绑定建议支持：

- 按统一社会信用代码自动匹配
- 按外部企业 ID 直接绑定
- 冲突人工确认
- 一企一主绑定
- 保留绑定历史

### 11.5 权限接口预留

需预留：

- 外部身份映射
- 单点登录
- 外部组织关系同步
- 用户角色初始化
- 数据范围继承

即使未来接入外部认证，平台内部仍应保留自己的业务授权模型。

### 11.6 同步模式

建议支持：

- 拉取同步
- 事件触发同步
- 人工触发同步

### 11.7 接口异常处理

建议具备：

- 同步失败重试
- 同步日志记录
- 冲突标记
- 人工确认覆盖
- 暂存最近成功结果
- 降级展示

### 11.8 历史快照原则

- 企业主档可被外部同步更新
- 审计项目快照不自动被外部覆盖
- 归档报告不因外部主数据变化而重写
- 历史审核记录不因外部账号变化而失真

## 12. 非功能需求与工程约束

建议将非功能需求纳入正式建设范围：

- 安全与权限控制
- 数据安全与留痕

## 14. SpreadJS 在线表格集成架构

### 14.1 集成定位

SpreadJS 是本平台数据采集层的核心技术选型，用于支撑复杂二维/多维数据录入场景。SpreadJS 在前端承担"类 Excel 交互体验"，但**不承担主流程控制、权限管理和审核闭环**。

### 14.2 集成边界

SpreadJS 适用范围：

- 24 个填报子模块中的表格型数据录入
- 多维数据交叉表（如能源品种 × 月份矩阵）
- 带公式的业务计算表（如能源平衡表、产品单耗表）
- 模板导入/导出（Excel 格式兼容）
- 数据批量编辑与粘贴

SpreadJS 不承担：

- 登录认证与权限控制
- 项目状态机流转
- 审核评分与整改闭环
- 报告版本管理
- 外部接口同步

### 14.3 架构方案

采用 **SpreadJS 适配器隔离架构**：

```text
apps/web/src/modules/data-entry/
  components/
    spreadsheet-adapter/        ← SpreadJS 封装层
      spreadsheet-host.tsx      ← SpreadJS 实例宿主组件
      workbook-initializer.ts   ← 工作簿初始化（模板加载）
      data-binder.ts            ← 业务数据 ↔ SpreadJS 双向绑定
      formula-bridge.ts         ← 公式注册与计算桥接
      event-handler.ts          ← SpreadJS 事件 → 业务事件转换
      export-handler.ts         ← 导出能力（Excel/PDF）
    form-renderer.tsx           ← 简单表单渲染器
    table-renderer.tsx          ← 原生表格渲染器（轻量场景）
    module-runner.tsx           ← 模块运行器（根据模块类型选择渲染器）
```

### 14.4 数据流设计

```text
[后端 DataRecord/DataItem] 
  → API 获取 → module-runner 
  → spreadsheet-adapter/data-binder (结构化数据 → SpreadJS 单元格映射) 
  → SpreadJS 展示与编辑 
  → data-binder (SpreadJS 变更 → 结构化数据提取) 
  → API 保存 → [后端 DataRecord/DataItem]
```

关键原则：

- 所有数据提取基于**稳定字段编码**（field code），不基于单元格坐标
- SpreadJS 内部公式仅用于前端即时预览，最终计算结果由后端 calculation-engine 确认
- 保存时提取的是结构化的 DataItem 列表，不是 SpreadJS 工作簿快照
- SpreadJS 工作簿模板与 TemplateVersion 绑定，历史项目使用历史模板

### 14.5 模板管理

- 每个填报模块对应一个 SpreadJS 工作簿模板
- 模板包含：列头定义、行结构、公式、验证规则、样式
- 模板按 TemplateVersion 版本化管理
- 管理端可上传/更新模板，发布新版本
- 企业端根据项目绑定的 TemplateVersion 加载对应模板

### 14.6 协同填报与锁机制

为避免多人同时编辑同一填报模块导致数据冲突：

- 每个 DataRecord 支持 `lockHolderId` 和 `lockAcquiredAt` 字段
- 用户打开填报模块时，尝试获取锁（乐观锁 + 超时自动释放）
- 锁超时时间建议为 30 分钟，超时后自动释放
- 其他用户打开同一模块时显示"只读"提示和当前编辑者信息
- 管理端可强制释放锁

### 14.7 离线与断点续填

- 所有录入数据在用户操作时实时暂存到浏览器 localStorage
- 网络恢复后自动同步到服务端
- 支持页面刷新后恢复到最后编辑状态
- 提交前校验服务端最新数据版本，冲突时提示用户

## 15. 碳排放因子管理

### 15.1 业务背景

碳排放计算是能源审计的重要组成部分。平台需要管理碳排放因子数据，支撑企业碳排放量的自动计算。

### 15.2 管理能力

管理端应提供：

- 碳排放因子的 CRUD 管理
- 按能源品种分类管理
- 支持国家标准默认值和自定义值
- 因子适用年份管理
- 氧化率参数管理
- 来源标准标注（如 GB/T 32150-2015）
- 因子版本历史

### 15.3 数据结构

碳排放因子对象包含：

- 能源品种编码
- 排放因子值（tCO2/单位）
- 氧化率
- 来源标准
- 适用年份
- 计量单位
- 是否默认值

### 15.4 计算集成

- 碳排放计算公式：`排放量 = 活动数据 × 排放因子 × 氧化率`
- 计算引擎从碳排放因子表获取参数
- 计算结果保存为 CalculationSnapshot
- 因子参数快照随计算快照一同保存，确保历史可追溯

## 16. 平台合并策略（节能诊断 + 能源审计）

### 16.1 合并背景

根据需求，节能诊断平台与能源审计平台需要合并为统一平台。两个平台在数据结构、业务流程上有大量重叠。

### 16.2 合并策略

建议采用**渐进式合并**：

- 阶段一：统一登录入口和企业档案
- 阶段二：统一主数据（能源品种、产品、单元定义）
- 阶段三：统一审核和整改流程
- 阶段四：统一报告模板和输出

### 16.3 技术预留

当前平台设计应预留以下能力：

- 业务类型标识：每个 AuditBatch 和 AuditProject 增加 `businessType` 字段（如 "energy_audit" / "energy_diagnosis"）
- 模板差异化：通过 TemplateVersion 区分不同业务类型的填报模板
- 模块启停：通过 ModuleConfig 控制不同业务类型的模块可见性
- 流程差异：通过状态机配置支持不同的流程分支
- 报告差异：通过 ReportTemplate 区分不同业务类型的报告格式

### 16.4 数据隔离

- 同一企业可同时参与能源审计和节能诊断
- 两类业务的填报数据独立存储（不同的 AuditProject）
- 共享企业主数据和配置数据
- 统计分析时可按业务类型筛选或汇总

## 17. 超期预警与截止日期管理

### 17.1 预警对象

以下业务对象支持截止日期和超期预警：

- AuditBatch：填报截止日期、审核截止日期
- AuditProject：项目整体截止日期
- RectificationTask：整改截止日期

### 17.2 预警机制

- 系统定时任务（BullMQ 调度）每日扫描即将超期和已超期的对象
- 超期前 N 天（可配置，建议 7 天、3 天、1 天）发送预警
- 超期后标记 `isOverdue = true`
- 预警通知发送到待办中心和消息通知

### 17.3 超期处理

- 超期不自动阻断业务（避免数据丢失）
- 管理端可查看所有超期项目并督办
- 超期项目在列表中醒目标识
- 支持管理端延期操作（修改截止日期并记录原因）

## 18. 数据回滚与版本管理

### 18.1 回滚场景

- 企业误操作提交后需要退回修改
- 管理端审核退回需要企业重新填报
- 数据导入错误需要批量回滚

### 18.2 回滚机制

- 填报数据支持"退回"操作（状态从 submitted → returned）
- 退回时记录退回原因和退回人
- 退回后企业可重新编辑并重新提交
- 关键数据操作保留审计日志
- 不支持任意历史版本回滚（过于复杂），仅支持状态退回

### 18.3 导入回滚

- 导入操作记录 ImportJob，包含导入前数据快照标识
- 导入失败时自动回滚到导入前状态
- 导入成功后可在一定时间内手动撤销（管理端操作）

## 19. 能源流程图与能效分析

### 19.1 能源流程图

能源流程图用于展示企业能源从输入到各用能单元的流向和消耗：

- 以 Sankey 图或流程图形式展示
- 数据来源：能源消耗数据 + 单元定义 + 产品定义
- 自动根据填报数据生成
- 支持导出为图片嵌入报告

### 19.2 能效对标分析

- 支持企业能效指标与行业标杆值对比
- 标杆值来源：国家标准、行业标准、历史最优
- 对标维度：综合能耗、产品单耗、设备能效
- 输出：对标分析表、差距分析图

### 19.3 技术实现

- 前端使用 ECharts 或类似图表库渲染 Sankey 图
- 后端 ChartOutput 对象存储图表配置和数据
- 图表配置通过 ChartConfig 定义（维度、指标、类型）
- 图表数据从 CalculationSnapshot 提取
